# Manipulator Arms (Serial-Chain)

> Conventions for serial-chain robot arms on ROS 2: URDF/SRDF, kinematics & dynamics,
> MoveIt 2 planning, joint-trajectory control, Cartesian/impedance control, and collision.

---

## Overview

This guide covers serial-chain manipulators (e.g. 6/7-DoF arms). The default stack is a
URDF/SRDF robot description, MoveIt 2 for motion planning, and `ros2_control`'s
`joint_trajectory_controller` (JTC) for execution. Dynamics/Jacobian math uses
[Pinocchio](https://github.com/stack-of-tasks/pinocchio) or KDL. Read
[Dynamics](../../dynamics.md) for frame, twist, and Jacobian sign conventions first.

---

## Robot description: URDF + SRDF

The URDF defines links, joints, **joint limits**, inertials, and collision geometry; the
SRDF (generated by the MoveIt Setup Assistant) defines planning groups, named poses, the
end effector, and the disabled-collision matrix.

```xml
<joint name="joint_2" type="revolute">
  <parent link="link_1"/> <child link="link_2"/>
  <axis xyz="0 1 0"/>
  <limit lower="-2.094" upper="2.094" effort="150.0" velocity="2.5"/>
</joint>
```

Use convex collision meshes (or primitives), not the visual meshes — full visual meshes
make collision checking orders of magnitude slower and brittle.

## Kinematics & dynamics

Keep an analytic or numerical IK solver behind a small interface. Forward kinematics,
the geometric Jacobian, and inverse dynamics come from Pinocchio:

```cpp
pinocchio::forwardKinematics(model, data, q);          // FK
pinocchio::computeJointJacobians(model, data, q);
Eigen::MatrixXd J(6, model.nv);
pinocchio::getFrameJacobian(model, data, ee_id,
    pinocchio::LOCAL_WORLD_ALIGNED, J);                 // 6 x n Jacobian
// Cartesian velocity: twist = J * qdot ; resolved-rate IK: qdot = J.pinv() * twist
```

Near a singularity `J` loses rank and `J.pinv()` blows up — use a damped least-squares
(Levenberg–Marquardt) pseudo-inverse and cap commanded joint velocity.

## MoveIt 2

Plan in a **planning group** (the kinematic chain base→tip). Default planners: OMPL
(sampling-based, general) or Pilz Industrial Motion Planner (deterministic LIN/PTP/CIRC
for predictable industrial paths). Always plan against an up-to-date **planning scene**.

```cpp
moveit::planning_interface::MoveGroupInterface arm(node, "manipulator");
arm.setMaxVelocityScalingFactor(0.3);
arm.setMaxAccelerationScalingFactor(0.3);
arm.setPoseTarget(target_pose);            // target in arm.getPlanningFrame()
moveit::planning_interface::MoveGroupInterface::Plan plan;
if (arm.plan(plan) == moveit::core::MoveItErrorCode::SUCCESS) arm.execute(plan);
```

Add obstacles via `PlanningSceneInterface` (collision objects) so the planner avoids
them. Velocity/acceleration scaling re-times the trajectory under joint limits — keep it
conservative for first runs.

## Execution: joint_trajectory_controller

MoveIt hands a `trajectory_msgs/JointTrajectory` to the JTC. The action server is
`/<controller>/follow_joint_trajectory`. The trajectory must be **time-parameterized**:
each point needs `time_from_start`, and positions/velocities/accelerations must respect
URDF limits, or the controller rejects or jerks.

```yaml
joint_trajectory_controller:
  ros__parameters:
    joints: [joint_1, joint_2, joint_3, joint_4, joint_5, joint_6]
    command_interfaces: [position]
    state_interfaces: [position, velocity]
    constraints:
      stopped_velocity_tolerance: 0.01
      goal_time: 0.5
```

The `joints` order here must match the order MoveIt sends; mismatched ordering silently
maps commands to the wrong joints.

## Cartesian & impedance control

For contact tasks (insertion, polishing, human interaction), pure position control is
too stiff. Use a Cartesian impedance/admittance controller: command a virtual
spring-damper about a target pose so the arm yields to external force.

```
F = K_p * (x_des - x) - K_d * x_dot      // task-space wrench
tau = J^T * F + g(q)                      // map to joint torques, add gravity comp
```

This requires a torque or effort command interface and accurate `g(q)` from inverse
dynamics. On a position-only arm, fall back to admittance control reading an F/T sensor.

## End effector & gripper

- Define the end-effector group and parent link in the SRDF; MoveIt plans to the EE
  frame, not the wrist.
- Drive grippers through their own controller/action (`GripperCommand`), never by
  jamming gripper joints into the arm trajectory.
- Account for the grasped payload: attach it as a collision object
  (`attachObject`) so planning treats it as part of the robot.

---

## Workspace, singularity & collision limits

- Respect URDF joint limits everywhere; clamp IK output before sending.
- Detect singularities via the Jacobian's smallest singular value; slow down or abort.
- Always enable self-collision and environment collision checking; keep the SRDF
  disabled-collision matrix limited to genuinely-adjacent links.

---

## Common mistakes

- **Ignoring joint limits.** Sending raw IK output past URDF `lower`/`upper`/`velocity`
  limits causes rejected goals, clipping, or protective stops.
- **No collision checking.** Executing a MoveIt-less straight-line joint command, or
  planning against a stale planning scene, drives the arm through obstacles.
- **Bad time parameterization.** A `JointTrajectory` with zero or inconsistent
  `time_from_start`, or velocities exceeding limits, makes the JTC jerk or reject it.
- **Joint-order mismatch.** The controller `joints` list not matching the planner's
  order maps commands to the wrong joints — dangerous and hard to spot.
- **Visual meshes as collision geometry.** Slow, unreliable collision checks; use convex
  hulls or primitives.
- **Singularity blow-up.** Naive `J.pinv()` near a singularity yields huge joint
  velocities; use damped least-squares and velocity caps.
- **Forgetting the payload.** Not attaching a grasped object lets the planner collide the
  payload with the world.
